## 1. Что это за проект

Веб-игра по типу «Монополии» для университетов. Преподаватель создаёт **изолированную
комнату** под свою группу и присылает студентам ссылку. Студенты ходят **по очереди** по
общему полю из **24 клеток**; **все видят ходы друг друга в реальном времени**. На клетках
задаются вопросы по предмету. Игра **пошаговая** с **авторитетным сервером**: вся логика и
подсчёт очков считаются на сервере, клиент только отображает состояние и шлёт намерения.

Преподаватель **не играет**, а выступает как **хост, модератор и наблюдатель**: создаёт банк
вопросов, запускает игру, ставит паузу, исключает игроков, вручную начисляет баллы, проверяет
экзаменационные ответы и получает итоговый отчёт с экспортом в CSV/Excel.

- **Студент** играет с телефона (mobile-first, вертикальная ориентация, без аккаунта).
- **Преподаватель** работает с десктопа/планшета (дашборд с аккаунтом).
- Язык интерфейса — русский.

Реализован **MVP (фазы 0–7)** из плана: каркас, банки вопросов с импортом, комнаты и
присутствие, ядро ходов, спец-клетки, модерация, устойчивость к отключениям, отчёты.
Фаза 8 (генерация вопросов ИИ) заложена архитектурно, но не реализована.

## 3. Архитектура и потоки данных

```
┌─────────────────────┐         WebSocket          ┌────────────────────────────┐
│  Студент (мобильный  │ ◄──── состояние комнаты ──► │       FastAPI сервер        │
│  браузер, React SVG) │   намерения (roll/answer)  │  ┌──────────────────────┐  │
└─────────────────────┘                            │  │  Игровой движок       │  │
                                                    │  │  (авторитет):         │  │
┌─────────────────────┐         WebSocket          │  │  - менеджер комнат    │  │
│ Преподаватель        │ ◄── модерация/очередь ───► │  │  - конечный автомат   │  │
│  (десктоп, дашборд)  │   exam review, pause...    │  │  - резолверы клеток    │  │
└─────────────────────┘                            │  │  - таймеры (asyncio)  │  │
        │  REST (auth, банки, отчёты)               │  └──────────┬───────────┘  │
        ▼                                           │   live-состояние (память)  │
┌─────────────────────┐                            └─────────────┬──────────────┘
│   FastAPI REST API   │                                         ▼
└──────────┬──────────┘                              ┌──────────────────┐
           ▼                                          │  PostgreSQL       │
   ┌───────────────┐                                  │  (постоянное:     │
   │  Экспорт CSV/  │                                  │  банки, комнаты,  │
   │  Excel         │                                  │  игроки, ответы,  │
   └───────────────┘                                  │  события)         │
                                                       └──────────────────┘
```

**Принцип авторитетности.** Клиент никогда не получает правильный ответ до резолва. Бросок
кубика, вращение колеса, проверка ответов, начисление баллов и переход хода считаются на
сервере и рассылаются всем в комнате (broadcast). Это одновременно анти-чит и механизм «все
видят чужой ход».

**Два слоя состояния:**
- **Постоянное** — в PostgreSQL (банки, вопросы, комнаты, игроки, ответы, журнал событий).
- **Живое (runtime)** — в памяти процесса: чей ход, текущая фаза, дедлайны таймеров, выданные
  вопросы. Хранится в `RoomManager`/`GameEngine`. При перезапуске сервера восстанавливается из
  БД (статус, очки, позиции), а активный ход возобновляется методом `ensure_running()`.

## 5. Бэкенд — разбор по файлам

### 5.1. Корень `backend/`

- **`requirements.txt`** — список Python-зависимостей с закреплёнными версиями.
- **`pyproject.toml`** — метаданные проекта; настройки `pytest` (`asyncio_mode = "auto"`,
  `testpaths = ["tests"]`) и `ruff` (длина строки 100).
- **`alembic.ini`** — конфигурация Alembic. Поле `sqlalchemy.url` намеренно пустое — URL
  берётся из `app.core.config` внутри `alembic/env.py`.
- **`.env.example`** — образец переменных окружения. Копируется в `.env`.
- **`sample_questions.csv`** — пример банка вопросов (физика) для проверки импорта.
- **`Dockerfile`** — образ на базе `python:3.12-slim`: установка зависимостей, запуск
  `uvicorn app.main:app`.

### 5.2. `app/core/` — ядро инфраструктуры

- **`config.py`** — класс `Settings` (Pydantic Settings). Читает из `.env`:
  `DATABASE_URL`, `SECRET_KEY`, `ALGORITHM`, `ACCESS_TOKEN_EXPIRE_MINUTES`,
  `FRONTEND_ORIGIN`, параметры ИИ-провайдера. Свойство `is_sqlite` определяет тип БД.
  Экземпляр `settings` кэшируется (`@lru_cache`).
- **`db.py`** — async-движок SQLAlchemy и фабрика сессий.
  - `Base` — декларативный базовый класс для всех моделей.
  - `engine` — `create_async_engine(...)`. Для SQLite добавляются `check_same_thread=False`
    и **`poolclass=NullPool`** (пул соединений плохо живёт с несколькими event-loop'ами,
    например в тестах — даёт «event loop is closed»). Для PostgreSQL — стандартный пул.
  - `SessionLocal` — `async_sessionmaker`.
  - `get_session()` — FastAPI-зависимость, выдающая сессию на время запроса.
- **`security.py`** — хеширование паролей (`hash_password`, `verify_password` на bcrypt) и
  JWT (`create_access_token`, `decode_access_token` на HS256).

### 5.3. `app/models/` — ORM-модели

- **`enums.py`** — все перечисления предметной области: `QuestionType` (zachet/exam),
  `QuestionDifficulty`, `QuestionSource` (manual/ai/ai_reviewed), `QuestionStatus`
  (draft/approved), `EndConditionType` (rounds/time/score), `RoomStatus`
  (lobby/playing/paused/finished), `AnswerStatus`, `CellType` (CORNER/ZACHET/EXAM/LOCATION),
  `CornerKind` (start/bet/wheel/skip), `LocationEffect` (free_points/timed_question/
  swap_with_previous/flat_penalty/flat_bonus). Все — `str, Enum` (значения сериализуются как
  строки).
- **`tables.py`** — таблицы SQLAlchemy 2.0 (типизированные `Mapped`). Типы выбраны
  **переносимо между PostgreSQL и SQLite**: `JSON` вместо `JSONB`, а enum'ы оформлены через
  хелпер `_enum()` (`sa.Enum(..., native_enum=False)` → хранится как `VARCHAR` + `CHECK`).
  Модели: `Teacher`, `QuestionBank`, `Question`, `Room`, `Player`, `Answer`, `GameEvent`
  (см. раздел 7).
- **`__init__.py`** — реэкспорт моделей и enum'ов, чтобы Alembic «видел» все таблицы.

### 5.4. `app/schemas/` — Pydantic-схемы

- **`auth.py`** — `TeacherRegister`, `TeacherLogin`, `TokenResponse`, `TeacherOut`.
- **`question.py`** — `QuestionBase` с валидатором по типу: для `zachet` обязательны 4–6
  вариантов и корректный индекс; для `exam` варианты обнуляются. `QuestionCreate`,
  `QuestionUpdate`, `QuestionOut`, `BankCreate`, `BankOut` (с подсчётом вопросов).
- **`room.py`** — `RoomCreate` (с ограничениями: `turn_timer_sec` 5–120, `answer_timer_sec`
  5–300, `max_players` 2–8), `RoomOut`, `RoomJoinInfo` (публичная инфа по коду).

### 5.5. `app/game/` — игровой движок (сердце проекта)

- **`board.py`** — генерация поля.
  - `DEFAULT_LAYOUT` — раскладка 24 клеток (углы на индексах 0/6/12/18).
  - `SCORING` — параметры начислений (зачёт +5, экзамен +10/−2, Старт +10, ставки 5–20 и т.д.).
  - `WHEEL_SECTORS` — 6 равновероятных секторов колеса фортуны.
  - `build_board_config()` — снимок раскладки + scoring + секторы, который записывается в
    `room.board_config` (игра не зависит от изменений дефолта в коде).
  - `next_exam_index()` — ближайшая клетка EXAM вперёд (для сектора «на экзамен»).
- **`state.py`** — рантайм-состояние в памяти (dataclasses).
  - `TurnPhase` — фазы хода (idle, awaiting_roll, awaiting_answer, awaiting_bet, awaiting_wheel).
  - `RuntimePlayer` — живое состояние игрока (очки, позиция, connected, пропуски, круги);
    метод `public()` отдаёт безопасный для клиента словарь.
  - `PendingAction` — контекст текущей ожидаемой фазы (вопрос/ставка/колесо), включая
    **правильный индекс ответа, который хранится только на сервере**, и `timer_token` для
    отмены устаревших таймеров.
  - `RoomRuntime` — состояние комнаты целиком: игроки, порядок ходов, текущий игрок, фаза,
    выданные вопросы; метод `snapshot()` строит полный `room_state` для broadcast.
- **`engine.py`** — `GameEngine`, авторитетный конечный автомат хода. Один экземпляр на
  активную комнату. Каждый публичный метод-действие защищён `asyncio.Lock`. Основное:
  - `ensure_running()` — **возобновление игры после перезапуска сервера**: если комната
    `playing`/`paused`, но активного хода нет (рантайм потерян), начинает новый ход.
  - `start_game()`, `_advance_turn()`, `_end_turn()` — управление очередью ходов.
  - `handle_roll()`, `_move()`, `_resolve_cell()`, `_resolve_corner()`, `_resolve_location()`,
    `_swap_with_previous()` — бросок и резолв клеток.
  - `_present_question()`, `handle_answer()`, `_record_answer()` — выдача вопроса и проверка.
  - `handle_spin()` (колесо), `handle_bet()`/`handle_decline_bet()` (ставка).
  - `_push_exam_queue()`, `resolve_exam()` — асинхронная проверка экзаменов преподавателем.
  - `pause()`/`resume()`, `kick_player()`, `adjust_score()`, `end_game()` — модерация.
  - `_check_end_condition()`, `_finish_game()`, `_compute_ranking()` — условия конца и рейтинг
    (с поддержкой нескольких первых мест при равенстве).
  - `_arm_timer()`/`_timeout_after()` — серверные таймеры на asyncio (источник истины), с
    токенами для отмены устаревших срабатываний.
  - `on_player_connected()`/`on_player_disconnected()` — присутствие.
- **`room_manager.py`** — `RoomManager`, реестр активных движков в памяти.
  - `get_engine()` — возвращает движок по коду; при первом обращении загружает комнату из БД
    (`_load`) и вызывает `ensure_running()` (возобновление хода после перезапуска).
  - `add_player()` — регистрация студента в лобби (создаёт запись `Player`, выдаёт
    `session_token`).
  - `find_player_by_token()` — поиск игрока по токену для реконнекта.

### 5.6. `app/ws/` — реальное время

- **`protocol.py`** — константы типов сообщений: `ClientMsg` (студент→сервер), `TeacherMsg`
  (преподаватель→сервер), `ServerMsg` (сервер→клиенты). Хелпер `msg(type, payload)`.
- **`connection.py`** — `ConnectionManager`: реестр WS-подключений по комнатам, отдельно
  студенты и преподаватели; методы `attach_player`, `attach_teacher`, `detach`, `send`,
  `send_to_player`, `broadcast`, `broadcast_teachers`. При повторном подключении игрока
  старый сокет закрывается.
- **`routes.py`** — два WebSocket-эндпоинта:
  - `/ws/play/{code}` — канал студента. Первое сообщение `join_room` или `rejoin`; далее
    `roll_dice`, `submit_answer`, `place_bet`, `decline_bet`, `spin_wheel`, `ack`.
  - `/ws/teacher/{code}?token=JWT` — канал преподавателя (аутентификация по JWT в query).
    Команды: `start_game`, `pause`, `resume`, `kick_player`, `adjust_score`, `resolve_exam`,
    `end_game`. Обрабатывает `WebSocketDisconnect`/`RuntimeError`, при разрыве вызывает
    `on_player_disconnected`.

### 5.7. `app/api/` — REST

- **`deps.py`** — зависимость `get_current_teacher` (распаковка JWT → объект `Teacher`).

## 8. WebSocket-протокол

Одно WS-подключение на клиента. Сообщения — JSON `{ "type": ..., "payload": ... }`.

### Студент → сервер
`join_room` `{code, name, surname, group}` · `rejoin` `{session_token}` · `roll_dice` ·
`submit_answer` `{answer}` · `place_bet` `{amount}` / `decline_bet` · `spin_wheel` · `ack`.

### Преподаватель → сервер
`start_game` · `pause` / `resume` · `kick_player` `{player_id}` · `adjust_score`
`{player_id, delta}` · `resolve_exam` `{answer_id, accepted}` · `end_game`.

### Сервер → клиенты (broadcast в комнату)
`joined` `{session_token, player_id}` · `room_state` (**полный снимок**: игроки, чей ход,
фаза, дедлайны) · `player_joined` / `player_left` / `player_reconnected` /
`player_eliminated` · `game_started` / `game_paused` / `game_resumed` · `turn_started` ·
`dice_rolled` · `pawn_moved` · `question_presented` (**без правильного ответа!**) ·
`answer_result` · `exam_pending_review` · `wheel_result` · `bet_requested` / `bet_placed` ·
`score_updated` · `game_finished` `{ranking}` · `exam_review_queue` (только преподавателю) ·
`error`.

## 9. Игровой движок и конечный автомат хода

**Состояния комнаты:** `lobby → playing ⇄ paused → finished`.

**Под-фазы активного хода** (один активный игрок за раз):

```
awaiting_roll (таймер 15 c по умолчанию)
   └─ roll_dice → dice_rolled → moving → pawn_moved (прошёл/встал на Старт → +10)
        └─ resolving_cell → ветвление по типу клетки:
              ZACHET   → awaiting_answer → авто-проверка → +5/0 → end_turn
              EXAM     → awaiting_answer → submit → pending_review;
                         ХОД ПЕРЕХОДИТ СРАЗУ. Преподаватель резолвит позже → +10/−2.
              LOCATION → по эффекту: free_points (+5/0), timed_question (10 c, +5/0),
                         swap_with_previous (обмен очками), flat_bonus/flat_penalty (±N)
              CORNER:  Старт (+10), Ставка (5–20, ×2/сгорание), Колесо (6 секторов 1/6),
                       Прогул (пропуск)
end_turn: сбросить пропуски действовавшего; проверить условие конца; следующий игрок.
```

- **Таймеры** — серверные (asyncio), с токенами для отмены устаревших срабатываний. Истечение
  любого таймера = действие неверно/пропуск, `consecutive_missed_turns += 1`; при значении > 3
  игрок **авто-исключается**.
- **Экзамен** резолвится **асинхронно**: ход переходит дальше сразу, баллы применяются в момент
  проверки преподавателем (комната не блокируется).
- **Победитель** — максимум баллов; при равенстве несколько первых мест. Баллы могут уходить в
  минус; выбывание только по 3 пропускам, не по счёту.
- **Возобновление после перезапуска** — `ensure_running()` восстанавливает активный ход, если
  в БД статус `playing`, но рантайм-состояние было потеряно.

## 13. Журнал проделанной работы и исправленные баги

Проект построен «с нуля» по [PLAN.md](PLAN.md). Реализованы фазы 0–7 (полный MVP). По ходу
ручного тестирования был найден и устранён ряд проблем — ниже для прозрачности.

1. **Подключение БД.** Настроен PostgreSQL 17 (база `monopoly`), сгенерирована и применена
   начальная миграция Alembic. Код моделей сделан переносимым (JSON, не нативные enum'ы),
   чтобы работать и на SQLite.

2. **Зависание подключения преподавателя/студента к комнате** — компонент застревал на
   «Подключение…». Причина: защитный флаг от двойного подключения вместе с dev-режимным
   StrictMode закрывал сокет и не переподключался. Исправлено переходом на корректный паттерн
   эффекта (подключение при монтировании, отключение при размонтировании).

3. **Шум `error reading bcrypt version`** — закреплён `bcrypt==4.0.1`, который `passlib`
   корректно читает.

4. **«Ожидание хода…» без активного игрока после перезапуска бэкенда.** Живое состояние хода
   хранится только в памяти; `uvicorn --reload` при правке любого файла перезапускал процесс и
   терял его, оставляя комнату `playing` без активного хода. Добавлен `GameEngine.ensure_running()`
   (вызывается из `room_manager.get_engine`), который возобновляет ход; `resume()` тоже
   восстанавливает потерянный во время паузы ход.

5. **Сессии двух студентов в одном браузере перезаписывались.** Токен сессии перенесён из
   `localStorage` (общий для вкладок) в `sessionStorage` (свой у каждой вкладки) и сохраняется
   надёжно (по коду комнаты, известному при подключении).

6. **Флаки тестов на SQLite** («database is locked» / «event loop is closed»). Для async SQLite
   включён `NullPool`; добавлена фикстура изоляции. На PostgreSQL это не влияет.

7. **Клики по ответам и кнопки «Принять/Отклонить» у преподавателя не срабатывали** (главный
   баг). Воспроизведён в реальном браузере: в сторе WebSocket оказывался `null`/мёртвым,
   поэтому `socket.send` молча ничего не отправлял. Причина — связка **React StrictMode**
   (двойной mount в dev) и **Vite HMR** (горячая замена кода во время игры) оставляла
   осиротевший сокет. Серверная логика была корректна (доказано тестом `test_answer_flow.py`).
   Исправления: убран `<React.StrictMode>` из `main.tsx`; добавлен `import.meta.hot.dispose` в
   сторе (закрытие старого сокета при HMR). После полной перезагрузки страницы бросок, ответ и
   проверка экзамена работают стабильно.

8. **UX-улучшения по запросу.** Добавлен индикатор «чей ход» (`TurnBanner`) для студентов и
   преподавателя; панель действий (включая кнопку «Бросить кубик») вынесена справа от поля на
   десктопе; счёт обновляется сразу по `score_updated`.

## 14. Известные ограничения и планы развития

- **Масштаб.** MVP рассчитан на один инстанс сервера с живым состоянием в памяти (много
  небольших комнат ≤ 8 игроков). Путь роста — вынести live-состояние в **Redis** (pub/sub) и
  запускать несколько инстансов.
- **Перезапуск во время игры** сбрасывает текущую фазу хода (восстанавливается с начала хода
  через `ensure_running`). Для полной устойчивости можно персистить фазу/таймеры в БД.
- **Фаза 8 — генерация вопросов ИИ.** Архитектура заложена (`app/ai/base.py`,
  `QuestionGenerator`, `QuestionDraft`), но провайдеры (GigaChat / OpenAI-совместимый /
  локальная модель) не реализованы. Требуют ключей API и UI-ревью черновиков.
- **Дедупликация вопросов по эмбеддингам** (опция из плана) не реализована.
- **HTTPS/WSS и продакшен-секреты** — для деплоя задать сильный `SECRET_KEY`, включить TLS,
  не логировать токены и правильные ответы (последнее уже соблюдается).
