# Образовательная игра-монополия для студентов — проектный план и архитектура

> Документ-бриф для Claude Code. Содержит полные правила, стек, архитектуру, модель данных,
> протокол реального времени, движок ходов, конвейер генерации вопросов и поэтапный план сборки.
> Язык интерфейса — русский. Студенты играют с телефона, преподаватель — с десктопа/планшета.

---

## 1. Краткое описание

Веб-игра по типу «Монополии» для университетов. Преподаватель создаёт **изолированную комнату**
под свою группу и присылает студентам ссылку. Студенты ходят **по очереди** по общему полю,
**все видят ходы друг друга в реальном времени**. На клетках задаются вопросы по предмету.
Игра **пошаговая** с авторитетным сервером (вся логика и подсчёт очков — на сервере, клиент только
отображает состояние и шлёт намерения).

Преподаватель **не играет**, а выступает как **хост, модератор и наблюдатель**.

## 2. Полные правила (источник истины для реализации)

### 2.1. Поле — 24 клетки

Поле — кольцо из 24 клеток. 4 угла + 20 промежуточных (8 «Зачёт», 4 «Экзамен», 8 «Локация»).

**Раскладка по умолчанию** (индексы 0–23, углы кратны 6; конфигурируема, но MVP использует эту):

| Индекс | Тип    | Подтип / локация |
|--------|--------|------------------|
| 0  | CORNER   | Старт |
| 1  | ZACHET   | — |
| 2  | LOCATION | Столовая |
| 3  | EXAM     | — |
| 4  | ZACHET   | — |
| 5  | LOCATION | Спортивный зал |
| 6  | CORNER   | Ставка на знания |
| 7  | ZACHET   | — |
| 8  | LOCATION | Библиотека |
| 9  | ZACHET   | — |
| 10 | EXAM     | — |
| 11 | LOCATION | Общежитие |
| 12 | CORNER   | Колесо фортуны |
| 13 | ZACHET   | — |
| 14 | LOCATION | Деканат |
| 15 | ZACHET   | — |
| 16 | EXAM     | — |
| 17 | LOCATION | Лаборатория |
| 18 | CORNER   | Прогул пар |
| 19 | ZACHET   | — |
| 20 | LOCATION | Кафедра |
| 21 | ZACHET   | — |
| 22 | EXAM     | — |
| 23 | LOCATION | Актовый зал |

Контроль: углы 4, ZACHET 8, EXAM 4, LOCATION 8 — итого 24.

### 2.2. Типы клеток и начисления

- **Зачёт (ZACHET).** Вопрос с **4–6 вариантами** ответа, проверка **автоматическая**.
  Верно → **+5**, неверно → **0**.
- **Экзамен (EXAM).** Более сложный вопрос **без вариантов** (открытый ответ).
  Проверяет **преподаватель вручную**. Верно → **+10**, неверно → **−2**.
- **Локация (LOCATION).** Клетка с условием (см. 2.4).

### 2.3. Угловые клетки

- **Старт.** **+10** баллов — и за **прохождение**, и за **остановку** (как в классической Монополии).
- **Ставка на знания.** Игрок выбирает ставку **5–20** баллов (минимум 5, максимум 20),
  **нельзя ставить больше, чем есть на счету**, **можно отказаться** (тогда обычный вопрос без ставки
  или пропуск эффекта — см. ниже). Задаётся вопрос. Верно → ставка **удваивается** (чистый прирост `+ставка`),
  неверно → ставка **сгорает** (`−ставка`).
- **Колесо фортуны.** Крутится колесо, **все 6 секторов равновероятны** (по 1/6):
  1. Получил повышенную стипендию → **+5**
  2. Закрыл сессию досрочно → **+10**
  3. Опоздал на экзамен → **−5**
  4. Выиграл олимпиаду → **+15**
  5. Отправляйся на ближайший экзамен → перемещение **вперёд** до ближайшей клетки EXAM и сразу ответ (**+10 / −2**)
  6. Поймали на списывании → **−10**
- **Прогул пар.** Пропуск хода (текущий ход завершается без действий).

### 2.4. Локации (эффекты — ПРЕДЛОЖЕННЫЕ значения по умолчанию, конфигурируемы)

Тип эффекта определяет поведение клетки:

| Локация        | Эффект (тип)         | Поведение |
|----------------|----------------------|-----------|
| Столовая       | `free_points`        | Лёгкий вопрос. Верно → +5, неверно → 0 (без штрафа). |
| Спортивный зал | `timed_question`     | Вопрос со строгим таймером **10 c**. Успел и верно → +5, иначе → 0. |
| Библиотека     | `free_points`        | Верно → +5, неверно → 0. |
| Общежитие      | `swap_with_previous` | Обмен суммарными баллами с **предыдущим игроком по очереди хода**. Без вопроса. |
| Деканат        | `flat_penalty`       | «Вызвали в деканат» → −3. Без вопроса. |
| Лаборатория    | `timed_question`     | Вопрос, таймер 10 c. Верно и вовремя → +5, иначе → 0. |
| Кафедра        | `free_points`        | Верно → +5, неверно → 0. |
| Актовый зал    | `flat_bonus`         | «Выступил на конференции» → +3. Без вопроса. |

> ⚠️ Значения и распределение эффектов — точка настройки. Вынести в конфиг комнаты/поля.

### 2.5. Кубик и ходы

- **Один кубик** (1–6). Бросок генерируется **на сервере**.
- **Таймер на ход — 15 секунд** (на то, чтобы бросить кубик / совершить действие).
- **Таймер на ответ** на вопрос — **задаёт преподаватель** при создании комнаты (общий для Зачёт/Экзамен).
  Локации с `timed_question` используют отдельный короткий таймер (по умолчанию 10 c).
- **Истёк любой таймер** → действие/ответ считается **неверным**, ход **автоматически завершается**.

### 2.6. Игроки, подключение, выбывание

- Игроков: **минимум 2, максимум 8** (максимум задаёт преподаватель в пределах 8).
- Студент входит по ссылке, вводит **имя, фамилию, название группы**. Аккаунт не нужен.
- **Отключение:** отключившийся игрок **пропускает ходы**, **может вернуться** (по session-токену).
- **Авто-исключение:** если игрок пропустил **более 3 ходов подряд** (т.е. на 4-й подряд пропуск) — **исключается автоматически** из ротации.

### 2.7. Условия окончания (преподаватель выбирает одно)

- **N кругов** (1–10), либо
- **Фиксированное время**, либо
- **Первый до N баллов**.
- Плюс преподаватель может **завершить игру в любой момент**.
- **Победитель — максимум баллов.** При равенстве — **несколько первых мест** (2+).
- Баллы **могут уходить в минус**; выбывание **только** по правилу 3 пропусков, не по счёту.

### 2.8. Преподаватель (не играет)

При создании комнаты задаёт: **предмет, банк вопросов, количество кругов (или иное условие конца),
таймеры, максимум игроков**.

Во время игры может: **поставить паузу**, **исключить игрока**, **вручную начислить/снять баллы**,
**принять спорный ответ** (в т.ч. ручная проверка экзаменационных ответов).

После игры получает отчёт: **итоговый рейтинг, число правильных ответов, процент успешности,
статистику по каждому вопросу, экспорт в CSV/Excel**.

## 3. Технологический стек

- **Frontend:** React + TypeScript + Vite. **Mobile-first** для студента (вертикальная ориентация,
  тач). Tailwind CSS. Поле — SVG (адаптивный масштаб под ширину экрана). Состояние — Zustand.
  Реальное время — нативный WebSocket-клиент с авто-реконнектом по session-токену.
- **Backend:** **Python FastAPI** (REST + WebSocket на Starlette). Авторитетный игровой движок.
  Pydantic v2 для схем и валидации.
- **БД:** PostgreSQL. ORM — SQLAlchemy 2.0 (async) + Alembic для миграций. (Локально допустим SQLite.)
- **Live-состояние:** в памяти процесса (для MVP, один инстанс). Путь масштабирования —
  Redis (pub/sub + хранение состояния комнат), **не требуется для MVP**.
- **Генерация вопросов (ИИ):** офлайн-модуль с **подключаемым провайдером** (GigaChat / YandexGPT /
  OpenAI-совместимый / локальная модель). Строгий JSON по Pydantic-схеме. Дефолт для РФ — GigaChat
  (есть бесплатный лимит). См. §9.
- **Экспорт отчётов:** openpyxl (Excel) + csv (CSV).
- **Контейнеризация:** docker-compose (postgres + backend + frontend).

> Альтернатива бэкенда — Node.js + Socket.IO (встроенные «rooms», авто-реконнект). Выбран FastAPI:
> единый язык с конвейером генерации и сильная сторона разработчика. Менять не рекомендуется без причины.

**Принцип авторитетности:** клиент никогда не получает правильный ответ до резолва. Бросок кубика,
вращение колеса, проверка ответов, начисление баллов и переход хода считаются на сервере и
рассылаются всем в комнате (broadcast). Это и анти-чит, и источник «все видят чужой ход».

## 6. Модель данных (PostgreSQL)

```sql
-- Преподаватели (единственная роль с аккаунтом)
teachers(id, email UNIQUE, password_hash, name, created_at)

-- Банки вопросов
question_banks(id, teacher_id FK, name, subject, created_at)

questions(
  id, bank_id FK,
  type ENUM('zachet','exam'),
  text,
  options JSONB NULL,            -- для zachet: массив из 4–6 строк
  correct_option_index INT NULL,-- для zachet
  reference_answer TEXT NULL,    -- для exam: эталон в помощь преподавателю
  explanation TEXT NULL,
  difficulty ENUM('easy','medium','hard') NULL,
  source ENUM('manual','ai','ai_reviewed'),
  status ENUM('draft','approved'),
  created_at
)

-- Комнаты (игровые сессии)
rooms(
  id, teacher_id FK, code UNIQUE,    -- code для входа по ссылке
  subject, bank_id FK,
  end_condition_type ENUM('rounds','time','score'),
  end_condition_value INT,            -- N кругов / секунды / N баллов
  turn_timer_sec INT DEFAULT 15,
  answer_timer_sec INT,               -- задаёт преподаватель
  max_players INT,                    -- ≤ 8
  board_config JSONB,                 -- снимок раскладки 24 клеток
  status ENUM('lobby','playing','paused','finished'),
  created_at, started_at, finished_at
)

-- Игроки в комнате
players(
  id, room_id FK,
  name, surname, group_name,
  session_token UNIQUE,               -- для реконнекта
  score INT DEFAULT 0,
  position INT DEFAULT 0,
  turn_order INT,
  connected BOOL DEFAULT true,
  consecutive_missed_turns INT DEFAULT 0,
  eliminated BOOL DEFAULT false,
  joined_at
)

-- Ответы (для статистики и модерации экзаменов)
answers(
  id, room_id FK, player_id FK, question_id FK,
  cell_index INT,
  given_answer TEXT,                  -- индекс варианта (zachet) или текст (exam)
  is_correct BOOL NULL,               -- NULL пока на ручной проверке
  points_delta INT,
  status ENUM('auto_correct','auto_wrong','pending_review','approved','rejected','timeout'),
  answered_at, resolved_at
)

-- Журнал событий (replay, анти-чит, статистика)
game_events(id, room_id FK, player_id FK NULL, type, payload JSONB, created_at)
```

**Выдача вопросов:** при попадании на Зачёт/Экзамен сервер берёт случайный `approved`-вопрос нужного
типа из банка комнаты, **не повторяя** уже выданные в этой игре; при исчерпании — разрешить повтор.
Банк должен содержать достаточно вопросов (валидировать при старте: предупредить, если мало).

## 7. WebSocket-протокол

Одно WS-подключение на клиента. Сообщения — JSON `{ "type": ..., "payload": ... }`.

### 7.1. Студент → сервер
- `join_room` `{code, name, surname, group}` → ответ `joined {session_token, player_id}`
- `rejoin` `{session_token}`
- `roll_dice` `{}`
- `submit_answer` `{answer}` — индекс варианта (zachet) или текст (exam)
- `place_bet` `{amount}` / `decline_bet` `{}`
- `spin_wheel` `{}` (или авто на сервере)
- `ack` `{}` — подтверждение анимации/готовности

### 7.2. Преподаватель → сервер
- `start_game` `{}`
- `pause` / `resume` `{}`
- `kick_player` `{player_id}`
- `adjust_score` `{player_id, delta}`
- `resolve_exam` `{answer_id, accepted: bool}`
- `end_game` `{}`

### 7.3. Сервер → клиенты (broadcast в комнату)
- `room_state` — **полный снимок**: игроки (позиции, очки, статус), чей ход, фаза, дедлайны таймеров
- `player_joined` / `player_left` / `player_reconnected` / `player_eliminated`
- `game_started` / `game_paused` / `game_resumed`
- `turn_started` `{player_id, deadline}`
- `dice_rolled` `{player_id, value}`
- `pawn_moved` `{player_id, from, to, passed_start: bool}`
- `question_presented` `{question_id, text, options?, cell_type, deadline}` — **без правильного ответа!**
- `answer_result` `{player_id, correct, points_delta}` — для авто-проверки (zachet/локации)
- `exam_pending_review` `{player_id}` — экзамен ушёл преподавателю
- `wheel_result` `{player_id, sector}`
- `bet_placed` `{player_id, amount}`
- `score_updated` `{player_id, score}`
- `game_finished` `{ranking}` — список с учётом возможных нескольких первых мест
- `error` `{code, message}`

### 7.4. Только преподавателю (его канал)
- `exam_review_queue` `{items: [{answer_id, player, question_text, reference_answer, given_answer}]}`
- Полная видимость состояния и эталонных ответов.

## 8. Игровой движок — конечный автомат хода

**Состояния комнаты:** `lobby → playing ⇄ paused → finished`.

**Под-фазы активного хода** (один активный игрок за раз):

```
awaiting_roll (таймер 15 c)
   └─ roll_dice → dice_rolled
        └─ moving → pawn_moved (если прошёл/встал на Старт → +10)
             └─ resolving_cell → ветвление по типу клетки:

   ZACHET   → awaiting_answer (answer_timer) → авто-проверка → +5/0 → end_turn
   EXAM     → awaiting_answer (answer_timer) → submit → answers(pending_review)
              → exam_pending_review; ХОД ПЕРЕХОДИТ ДАЛЬШЕ СРАЗУ (не блокируем комнату).
              Преподаватель резолвит позже → +10/−2 применяется в момент resolve.
   LOCATION → по эффекту:
              free_points → awaiting_answer → +5/0
              timed_question → awaiting_answer (10 c) → +5/0
              swap_with_previous → мгновенный обмен очками с пред. игроком → end_turn
              flat_bonus/flat_penalty → ±N → end_turn
   CORNER:
     Старт  → +10 (за остановку; за прохождение начислено на этапе moving) → end_turn
     Ставка → awaiting_bet (5–20, ≤ score, можно decline)
              → awaiting_answer → верно: +ставка, неверно: −ставка → end_turn
     Колесо → spinning → сектор (1/6) → применить эффект; если «ближайший экзамен» →
              переместить вперёд до ближайшего EXAM → awaiting_answer (exam, pending_review)
     Прогул → end_turn (без действий)

end_turn:
   - сбросить consecutive_missed_turns активного игрока в 0 (если он действовал)
   - проверить условие конца игры (кругов/время/баллы)
   - выбрать следующего не выбывшего игрока → awaiting_roll
```

**Таймауты:** истечение `awaiting_roll` или `awaiting_answer` → действие = неверно/пропуск,
`consecutive_missed_turns += 1`; при значении > 3 → `eliminated = true`. Отключённый игрок
автоматически «пропускает» свои awaiting-фазы по таймеру.

**Прохождение Старта:** засчитывать +10, если в ходе `moving` индекс пересёк 0 (новый круг).

**Подсчёт кругов:** круг засчитывается игроку при пересечении/попадании на Старт; условие `rounds`
сверяется по числу пройденных кругов.

## 9. Конвейер генерации вопросов (ИИ) — гибрид

Три источника в одну таблицу `questions` (`source`, `status`):

1. **Ручной ввод/загрузка** преподавателем (форма + импорт; см. форматы ниже) → сразу `approved`.
2. **Генерация ИИ (офлайн):** преподаватель указывает тему, тип (zachet/exam), количество, сложность
   → бэкенд вызывает LLM с требованием **строгого JSON-массива** → валидация по Pydantic →
   запись как `source='ai', status='draft'`.
3. **Ревью:** преподаватель просматривает черновики, **редактирует/одобряет/отклоняет** →
   одобренные становятся `source='ai_reviewed', status='approved'`.

В игру попадают **только `approved`**.

**Интерфейс провайдера** (`app/ai/base.py`):
```python
class QuestionGenerator(Protocol):
    def generate(self, topic: str, qtype: Literal['zachet','exam'],
                 count: int, difficulty: str) -> list[QuestionDraft]: ...
```
Реализации: GigaChat (дефолт для RU, есть бесплатный лимит), OpenAI-совместимая, локальная модель
(Qwen2.5 / T-lite — батч-генерация на бесплатном GPU, нулевая стоимость). Выбор провайдера — через конфиг.

**JSON-схема ответа модели** (валидируется, кривое отбраковывается):
```json
{
  "type": "zachet|exam",
  "text": "...",
  "options": ["...", "...", "...", "..."],   // только zachet, 4–6
  "correct_option_index": 0,                  // только zachet
  "reference_answer": "...",                  // только exam
  "explanation": "...",
  "difficulty": "easy|medium|hard"
}
```
Доп.: дедуп по эмбеддингам (опционально), тег сложности.

**Импорт ручных вопросов** (MVP): поддержать CSV/JSON-загрузку и форму. Пример CSV-колонок:
`type, text, option1..option6, correct_index, reference_answer, explanation, difficulty`.

> Примечание по стоимости (ориентир 2026): предгенерация банка — разовая и дешёвая
> (тысячи вопросов = рубли/бесплатный лимит), т.к. **не выполняется в рантайме**. On-the-fly не использовать.

## 10. Отчёты и экспорт

Источник — `answers` + `players` + `game_events`. По завершении игры (или по запросу) формировать:
- **Итоговый рейтинг** (с учётом нескольких первых мест при равенстве).
- **Число правильных ответов** на игрока.
- **Процент успешности** на игрока.
- **Статистика по каждому вопросу** (сколько ответили верно/неверно, средняя сложность по факту).
- **Экспорт в CSV и Excel** (openpyxl): кнопка скачивания в дашборде преподавателя.

## 11. Аутентификация и вход

- **Преподаватель:** аккаунт (email + пароль, JWT). CRUD банков/вопросов, создание комнат, модерация, отчёты.
- **Студент:** **без аккаунта**. Переходит по ссылке `…/join/{room_code}`, вводит имя, фамилию, группу.
  Сервер выдаёт `session_token` (хранится в памяти клиента) для реконнекта.

> Безопасность: пароли/токены не логировать; правильные ответы не отправлять клиенту до резолва;
> session_token — единственный секрет студента, передаётся по WSS.

## 12. Требования к интерфейсу

- **Студент — mobile-first.** Вертикальная ориентация, крупные тач-цели, поле-кольцо 24 клеток
  адаптивно вписано в ширину экрана (SVG, viewBox-масштаб). Видны: поле с фишками, чей ход и таймер,
  своя позиция/очки, карточка вопроса, кубик, колесо, панель ставки.
- **Преподаватель — десктоп/планшет.** Дашборд: банки вопросов, генерация ИИ + ревью черновиков,
  создание комнаты, лобби, **очередь проверки экзаменов**, кнопки модерации (пауза, кик, ±баллы),
  отчёт + экспорт.
- Реалтайм-индикаторы: подсветка активного игрока, обратный отсчёт таймера, анимация перемещения фишки,
  анимация колеса.
- Реконнект: при обрыве WS — авто-переподключение по `session_token`, восстановление по `room_state`.

---

## 13. Нефункциональные требования

- **Масштаб:** много небольших комнат (каждая ≈ группа, ≤ 8 игроков). MVP — один инстанс, состояние
  в памяти. Путь роста: вынести live-состояние в Redis + pub/sub, запускать несколько инстансов.
- **Надёжность таймеров:** серверные таймеры (asyncio), не полагаться на клиентские.
- **Идемпотентность действий:** игнорировать действия не от активного игрока и в неподходящей фазе.
- **Журналирование:** все значимые действия — в `game_events` (для отчётов и разбора спорных ситуаций).

## 14. План сборки по фазам (порядок реализации)

**MVP = Фазы 0–7** (полная игра + ручная загрузка вопросов + отчёты). Фаза 8 (генерация ИИ) — сразу
после MVP (архитектура уже заложена).

- **Фаза 0 — каркас.** Репозиторий, docker-compose, Postgres, Alembic, FastAPI-скелет, React+Vite+Tailwind,
  аутентификация преподавателя (JWT), базовый дашборд.
- **Фаза 1 — банки вопросов.** Модель `questions`/`question_banks`, CRUD, ручная загрузка (форма + CSV/JSON импорт),
  валидация (zachet: 4–6 вариантов + индекс; exam: эталон).
- **Фаза 2 — комнаты и присутствие.** Создание комнаты (параметры из §2.8), генерация `code`/ссылки,
  вход студента (имя/фамилия/группа), WS-подключение, лобби, broadcast `room_state`, реконнект по токену.
- **Фаза 3 — ядро ходов.** Конечный автомат, один кубик, перемещение, Старт (+10 проход/остановка),
  Зачёт (MCQ авто-проверка, +5/0), таймеры хода (15 c) и ответа, переход хода, условия конца (кругов/время/баллы),
  определение победителя (с несколькими первыми местами).
- **Фаза 4 — спец-клетки.** Экзамен (+ очередь ручной проверки, async-резолв, +10/−2), Ставка на знания
  (5–20, ≤ score, отказ, ×2/сгорание), Колесо фортуны (6 секторов 1/6, «ближайший экзамен»),
  Прогул пар, Локации (все эффекты из §2.4).
- **Фаза 5 — модерация преподавателя.** Пауза/resume, кик, ручные ±баллы, приём спорного ответа,
  очередь проверки экзаменов в реальном времени.
- **Фаза 6 — устойчивость.** Отключение/реконнект, пропуск ходов отключённого, авто-исключение
  при >3 пропусках подряд, корректные таймауты.
- **Фаза 7 — отчёты.** Агрегация статистики, рейтинг, % успешности, статистика по вопросам, экспорт CSV/Excel.
- **Фаза 8 — генерация ИИ (post-MVP).** Провайдер-интерфейс + GigaChat, строгий JSON + валидация + дедуп,
  UI ревью/редактирования черновиков, перевод в `approved`.
- **Фаза 9 — полировка.** Mobile-first доводка поля, анимации (фишка, колесо, кубик), индикаторы таймеров.

## 15. Принятые допущения (подтвердить/скорректировать)

1. Баллы могут уходить в минус; выбывание только по 3 пропускам подряд, не по счёту.
2. Экзамен резолвится **асинхронно** через очередь преподавателя — ход переходит дальше сразу,
   баллы (+10/−2) применяются в момент проверки (не блокируем комнату).
3. Раскладка поля и эффекты локаций (§2.1, §2.4) — **предложенные значения**, вынесены в конфиг.
4. Вопросы в пределах одной игры выдаются **без повтора** до исчерпания, затем повтор разрешён.
5. Таймер ответа — один на комнату для Зачёт/Экзамен; локации `timed_question` — отдельные 10 c.
6. «Отказ от ставки» = ход завершается без вопроса (эффект клетки не применяется).
7. Колесо «ближайший экзамен» — перемещение **только вперёд** до ближайшей клетки EXAM.

> Если по любому пункту решение иное — поправить соответствующий раздел перед стартом Фазы 3–4.
